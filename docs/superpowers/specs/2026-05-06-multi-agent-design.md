# Multi-Agent 合同校审架构设计

## 1. 概述

在现有单 Agent 架构基础上，引入多智能体团队协作模式，将合同校审和对话能力拆分为两个独立 Agent 团队，通过 Gateway 路由 + 三层记忆实现灵活编排与 Token 优化。

### 目标

- 将现有单 Agent 拆分为专业化分工的多 Agent 团队
- 通过流水线式编排提高审核质量
- 通过三层记忆和上下文压缩控制 Token 成本
- 通过可视化让 Agent 流转透明可追踪

## 2. 整体架构

```
Frontend (React) → SpringBoot Backend → gRPC
                                           ↓
                                    Gateway Router
                                    │    │    │
                                    │    │    └── 单 Agent 路径（快速模式）
                                    │    │         ↓
                                    │    │    SingleAgentHandler（现有 LangChain）
                                    │    │
                                    │    └─────── 多 Agent（自动）— 动态决策
                                    │
                                    └──────────── 多 Agent（手动）— 完整流水线
                                              /            \
                           审核团队 (LangGraph)   对话团队 (LangGraph)
                                   ↓                    ↓
                            ┌──────────────┐   ┌──────────────┐
                            │ 解析 Agent   │   │ QA Agent     │
                            │ 风险审查 Agent│   │ 追问澄清 Agent│
                            │ 法条引用 Agent│   │ 法理解释 Agent│
                            │ 改写建议 Agent│   │ 对比 Agent   │
                            │ 汇总 Agent   │   │ 深度研究 Agent│
                            └──────────────┘   └──────────────┘
                                   ↓                    ↓
                            ┌──────────────────────────────────────┐
                            │          共享记忆层                  │
                            │  Redis(热) → PG(温) → Milvus(冷)    │
                            └──────────────────────────────────────┘
```

### 2.1 Gateway Router

一个轻量级分类器，负责将用户请求路由到合适的 Agent 团队。

- **输入**：用户请求 + 当前上下文（contract_id、是否有进行中的审核）
- **输出**：`{ team: "review" | "dialogue", confidence, context_ids[] }`
- **Token 成本**：极低（每次路由 <100 tokens）
- **跨团队引用**：当对话需要引用审核结果时，自动从 PG 拉取相关审核发现注入对话上下文

Gateway 路由设计：

| 场景 | 路由 | 说明 |
|------|------|------|
| 明确审核请求（"帮我审一下这个合同"） | → 审核团队 | 启动新流水线 |
| 明确对话请求（"第8条是什么意思？"） | → 对话团队 | 按 QA 分类处理 |
| 模糊请求（"你觉得呢？"） | → 对话团队追问澄清 | 澄清后再判断 |
| 混合意图（"这个条款有问题吗？和之前版本比呢？"） | → 主导意图 | 首版按主导意图路由，辅意图由对话的 Specialist 处理 |
| 对话引用审核结果 | → 对话团队 + 注入审核结果 | Gateway 自动追加审核 findings 到对话上下文 |

### 2.2 单 Agent / 多 Agent 模式切换

用户可选择 Agent 协作模式，Gateway 根据选择调整路由逻辑。

#### 2.2.1 模式定义

| 模式 | 路由逻辑 | 适用场景 | Token 成本 |
|------|----------|----------|-----------|
| **单 Agent** | Gateway 直接路由到单个 LLM 调用（复用现有 LangChain 实现），跳过流水线编排 | 简单合同速查、快速问答、新用户上手 | 低（单次 LLM 调用） |
| **多 Agent（自动）** | Gateway 根据请求复杂度自动决定走单 Agent 还是多 Agent 流水线 | 默认模式，兼顾速度和深度 | 中-高（按需激活流水线） |
| **多 Agent（手动）** | 强制走完整多 Agent 流水线 | 复杂合同审查、合规关键场景、用户明确要求深度分析 | 高（完整流水线） |

#### 2.2.2 自动决策规则

多 Agent（自动）模式下，Gateway 根据以下信号决定路由：

| 信号 | 单 Agent 倾向 | 多 Agent 倾向 |
|------|--------------|--------------|
| 合同长度 | < 20 条款 | > 50 条款 |
| 问题复杂度 | 事实性提问（"第几条说了什么"） | 分析性提问（"是否合规"、"风险在哪"） |
| 历史交互 | 用户从未要求过深度分析 | 用户之前选择过"详细模式" |
| 系统负载 | 高负载时段 | 低负载时段 |
| 用户明确指定 | "简单看一下" | "详细审查"、"全面分析" |

#### 2.2.3 前端交互

- 审核入口处提供模式选择器（开关或下拉）：**快速** / **标准** / **深度**
  - 快速 → 单 Agent
  - 标准 → 多 Agent（自动）
  - 深度 → 多 Agent（手动）
- 可视化视图根据模式自适应：
  - 单 Agent：显示简洁的"处理中..."状态，无流水线详情
  - 多 Agent：显示完整 DAG/时间线流水线

#### 2.2.4 架构影响

两种模式共享同一套存储层（PG、Redis、Milvus），差异仅在 Gateway 的路由决策：

- **单 Agent 路径**：`Gateway → SingleAgentHandler (现有 LangChain 调用)`
- **多 Agent 路径**：`Gateway → LangGraph Pipeline → 多 Agent 编排`

结果统一写入 PG，可视化组件根据 `pipeline_mode` 字段渲染不同视图。

#### 2.2.5 模式切换约束

- 流水线运行时不可切换模式（需等待当前请求完成）
- 同一合同会话内可切换模式（不同请求用不同模式）
- 切换无需重新解析合同（解析结果在 PG 中共享）

## 3. Agent 团队

### 3.1 审核团队 (Review Team)

确定性流水线，5 个 Agent 顺序执行，含条件分支和容错路由。

**工作流：**

```
解析 Agent → 风险审查 Agent ─┬─ high_risk → 法条引用 Agent(优先)
                              └─ low_risk → 法条引用 Agent(正常)
                                     → 改写建议 Agent → 汇总 Agent → 完成
```

**Agent 职责：**

| Agent | 输入 | 输出 | 关键/可跳过 |
|-------|------|------|------------|
| 解析 Agent | 原始合同 | 结构化 clauses 列表 | 关键——解析失败则中止 |
| 风险审查 Agent | clauses | risk_findings[] | 关键——审查失败则中止（重试1次） |
| 法条引用 Agent | findings + clauses | legal_refs[] | 可跳过——无法条引用也可继续 |
| 改写建议 Agent | findings + refs + clauses | redraft_suggestions[] | 可跳过——无改写也可汇总 |
| 汇总 Agent | 所有上游输出 | summary_report | 关键——必须执行 |

**Agent 通信协议（结构化 JSON 手递）：**

```json
{
  "agent_id": "risk_checker",
  "status": "success",
  "input_summary": "审查了第 8 条赔偿条款",
  "findings": [
    { "clause": "8.1", "risk": "high", "summary": "赔偿上限为合同金额 10%", "suggestion": "建议调整为 30%" }
  ],
  "next_agent_hints": { "focus_clauses": ["8.1", "8.3"], "urgent": true }
}
```

### 3.2 对话团队 (Dialogue Team)

动态路由，先分类再按需激活 Specialist Agent。

**工作流：**

```
QA Agent → 分类判断 ─┬─ simple_qa → 最终回答(不触发 Specialist)
                      ├─ needs_clarify → 追问澄清 Agent → 重新分类
                      ├─ legal_explain → 法理解释 Agent → 最终回答
                      ├─ compare → 对比 Agent → 最终回答
                      └─ research → 深度研究 Agent → 最终回答
```

**分类路由规则：**

| 类型 | 典型问题 | 路由 |
|------|----------|------|
| simple_qa | "第几条说了什么？" | 直接返回，不用 Specialist |
| needs_clarify | "你觉得呢？"（模糊） | 追问澄清 → 重新路由 |
| legal_explain | "这条符合民法典吗？" | 法理解释 Agent |
| compare | "和之前版本有什么不同？" | 对比 Agent |
| research | "类似案件判决倾向？" | 深度研究 Agent |

**Token 优化：**
- ~70% 的问题只需 simple_qa，不触发 Specialist
- Specialist 可并行执行（复合问题）
- 先查 PG 温层再查 Milvus 冷层

## 4. 三层记忆系统

借鉴 Claude Code 的冷/温/热三层记忆架构，结合合同校审场景定制。

### 4.1 层级定义

| 层级 | 存储 | 内容 | 生命周期 | Token 成本 |
|------|------|------|----------|-----------|
| 🔥 热层 | Redis | 当前流水线执行状态 + 最近 3 轮对话完整上下文 | 流水线完成或闲置 30min 过期 | 0（不进 LLM context，纯控制流） |
| 💾 温层 | PostgreSQL | 当前合同会话的结构化输出 + 最近 10 轮对话摘要 | 从打开合同到归档 | 低（按 ID 按需读取） |
| 🧊 冷层 | Milvus | 历史对话/审核记录向量化 + 法规知识库 | 持久 | 中（语义检索 Top-K） |

### 4.2 热层边界

- **审核团队**：当前一轮流水线 5 个 Agent 的输入/输出，运行完释放
- **对话团队**：最近 3 轮 Q&A 的完整上下文（结构 + 自然语言）
- **控制流**：Pipeline 进度、已执行/待执行 Agent 列表

3 轮对话估算：约 1k-3k tokens（含用户问题 + Agent 中间结果 + 最终回复）

### 4.3 温层边界

- **审核结果**：完整的结构化输出存入 PG，按 agent_id + clause_id 精确读取
- **对话历史**：最近 10 轮 Q&A 的结构化记录（问题 + 答案摘要）
- **合同数据**：当前合同的解析结果、条款列表

### 4.4 冷层边界

- 超过 10 轮的对话 → 向量化存入 Milvus
- 过往合同的审核报告 → 语义检索
- 法律法规库（已有）→ 始终在 Milvus 中

**温→冷滑动策略：**
- 第 11 轮进入时，最早轮次的完整数据移至冷层
- 但该轮次的摘要保留在温层（"用户问过赔偿条款"→ chunk_id）
- 话题回溯时通过摘要指针 + 语义检索恢复上下文

## 5. 一致性模型

PG 为唯一可信源 (SSOT)，Redis 为缓存，Milvus 为异步索引。

### 5.1 写入路径

```
Agent 产出 → Redis (热层) → PG redo log → 流水线完成标记 → 异步写 Milvus
```

- 每个 Agent 完成后**立即写入 PG redo log**（非等流水线跑完）
- PG 写入失败 → 整个流水线回滚
- 流水线完成标记写入失败 → 后台检测 + 自动补标记

### 5.2 一致性保证

| 场景 | 保证 | 机制 |
|------|------|------|
| Redis 宕机 | 热层丢失，可从 PG 恢复 | Agent 从 PG redo log 恢复流水线，降级为跳过 Redis 直接读写 PG |
| PG 写入失败 | 流水线回滚 | 事务提交前任一写失败 → 整体回滚，Redis 不持久化 |
| Milvus 写入失败 | 温层不受影响，冷层最终一致 | PG 数据保留，后台重试队列（最多 3 次） |
| 对话引用刚完成的审核 | read-your-writes | 流水线完成标记确保数据已提交，对话从 PG 读取 |
| 用户并发操作多个合同 | 隔离 | 按 contract_id + pipeline_id 隔离，各自独立事务 |
| Redis 重建 | 缓存可重建 | 从 PG 读取最近 N 条记录回填 Redis，TTL 控制过期 |

### 5.3 推荐：Redo Log 模式

Agent 产出后立即写一条 PG redo log（含完整 payload），Redis 丢失时从 redo log 恢复。以最小额外开销（一条 PG 记录）换来热层的完全可恢复性。

## 6. 上下文压缩机制

长会话中自动检测上下文水位并渐进压缩。

### 6.1 水位检测

| 水位 | 触发时机 | 动作 |
|------|----------|------|
| 🟢 < 60% | — | 正常 |
| 🟡 60%-75% | 单次 Agent 调用后 | 轻度压缩：Agent 详细输出 → 结构化摘要 |
| 🟠 75%-90% | 每轮对话后 | 中度压缩：移除中间 Agent 输出，只留最终回答；最早轮次合并为摘要 |
| 🔴 > 90% | 立即 | 强制压缩：只留最近 2 轮完整，之前全部压缩为一段摘要 |

### 6.2 压缩后上下文结构

```
[System Prompt] [Agent 指令] [Session Summary]
━━━ 会话摘要（第 1-13 轮）━━━
• 用户询问了赔偿条款，建议上限从 10% 调至 30%
• 用户追问违约责任，识别到 3 个风险点
• ...
━━━ 第 14 轮（摘要）━━━
Q: 保密条款需要修改吗？ A: 发现保密期限过短...
━━━ 第 15 轮（完整）━━━
Q: 知识产权归属怎么约定的？ ...
```

### 6.3 关键原则

1. **压缩≠删除** — 完整数据始终在冷层可检索
2. **摘要附带 chunk_id 指针** — Agent 需要细节时精确定位
3. **渐进式压缩** — 每次过阈值只压缩最早的一部分
4. **异步执行** — 用户收到当前回答后，后台完成压缩，下轮生效
5. **用户可见** — 压缩时在回答末尾标记（"📦 上下文已压缩，历史细节可追问"）

## 7. 边缘情况与应对

### 7.1 流水线执行

| 边缘情况 | 风险 | 应对 |
|----------|------|------|
| Agent 超时 | 中 | 每个 Agent 设 30s 超时，重试 1 次，仍失败标记 failed |
| Agent 返回空结果 | 低 | 空 findings[] 是有效输出，汇总 Agent 能处理 |
| 用户中途取消 | 中 | 检查 cancellation token，已完成输出保留在 PG |
| Agent 幻觉（引用不存在条款） | 高 | 下游 Agent 校验 clause_id 是否在 clause_list 中 |
| Agent 输出超长 | 中 | 自带摘要标记，下游按需读取完整版 |
| 流水线整体超时 | 高 | 全局 5min 超时，已完成输出保留 |

### 7.2 热层 (Redis) 异常

| 边缘情况 | 风险 | 应对 |
|----------|------|------|
| 连接中断 | 高 | 从 PG redo log 恢复，降级为跳过 Redis |
| 内存 eviction | 中 | noeviction + TTL 30min，PG 有完整副本 |
| TTL 在流水线中过期 | 高 | 活跃流水线自动续期，闲置 30min 才允许过期 |

### 7.3 温层 (PostgreSQL) 异常

| 边缘情况 | 风险 | 应对 |
|----------|------|------|
| 连接池耗尽 | 高 | 等待队列 + 超时，超时后降级只读冷层 |
| redo log 写入失败 | 高 | 该 Agent 标记未持久化，Redis 宕机时需重新执行 |
| 流水线完成标记失败 | 中 | 后台检测"数据完整但未完成"的 pipeline，自动补标记 |

### 7.4 冷层 (Milvus) 异常

| 边缘情况 | 风险 | 应对 |
|----------|------|------|
| 服务不可用 | 中 | 降级为 PG 关键词搜索（pg_trgm），语义检索不可用但功能不中断 |
| Embedding API 超时 | 中 | 写入重试；读取时降级关键词搜索 |
| 部分写入失败 | 低 | 温层始终有完整数据，写入 job 幂等可重试 |

### 7.5 Gateway 路由异常

| 边缘情况 | 风险 | 应对 |
|----------|------|------|
| 意图不明确 | 中 | 返回 ambiguous 标记，降级为用户选择或对话团队处理 |
| 对话引用不存在的审核 | 中 | 返回"审核尚未完成/不存在"提示 |
| 混合意图（同一消息含审核和对话） | 高 | 首版按主导意图路由，辅意图由对话处理 |
| 用户切换合同 | 中 | Gateway 传递 active_contract_id，Agent 显式携带 |

### 7.6 会话与 Token 预算

| 边缘情况 | 风险 | 应对 |
|----------|------|------|
| 用户空闲后返回，热层已过期 | 中 | 从 PG 恢复最近 3 轮摘要到热层 |
| gRPC 连接断开 | 高 | 自带重试 + 指数退避，重试 3 次后返回"服务不可用" |
| Python Agent 重启 | 中 | 运行中的流水线标记 interrupted，用户可选择继续或重跑 |
| 超大合同（100+ 条款） | 高 | 解析 Agent 分块摘要，后续 Agent batch 处理，汇总合并 |
| 流水线重跑 | 中 | 跳过输入解析阶段，复用上次 PG 解析结果 |
| 累积预算耗尽 | 中 | 停止冷层检索，仅用温层结构化数据；摘要长度减半 |

### 7.7 安全

| 边缘情况 | 风险 | 应对 |
|----------|------|------|
| 合同内容 Prompt 注入 | 高 | 合同内容作为 data 输入（非 instruction），Agent 的 system prompt 明确约束 |
| 用户越权 | 高 | 权限由 SpringBoot 层处理，Agent 层拒绝无 token 请求 |
| Agent 输出泄露敏感信息 | 高 | SpringBoot 过滤层确保返回前移除合同外敏感内容 |

## 8. 可视化设计

采用混合视图（Option C），默认紧凑时间线可展开为 DAG 节点图。

### 8.1 数据流

```
Agent (Python) → Redis Pub/Sub → SpringBoot → WebSocket → 前端 (React)
     ↓                                                         ↓
  PG (持久化)                                             渲染 DAG/时间线
     ↓                                                         ↓
  Redis (缓存实时状态)                                   用户可点开每个 Agent 查看详情
```

### 8.2 交互设计

- **实时状态**：🟡 等待 → 🔵 运行中 → 🟢 完成 / 🔴 失败
- **耗时展示**：每个 Agent 显示执行耗时
- **点击展开**：点击节点查看输入摘要和输出 findings
- **错误高亮**：失败的 Agent 红色高亮，悬浮显示错误原因
- **对话轨迹**：对话团队展示路由路径（simple_qa / legal_explain / ...）
- **紧凑/展开切换**：默认紧凑时间线，点击展开为 DAG 节点图

## 9. 实施建议

### 9.1 优先级顺序

1. **审核团队先行** — 实现 5 Agent 流水线 + LangGraph 编排 + PG 持久化
2. **对话团队跟进** — 实现动态分类路由 + 按需激活 Specialist
3. **三层记忆** — Redis 热层 + PG 温层先行，Milvus 冷层后续
4. **上下文压缩** — 在长会话场景出现后迭代
5. **可视化** — 在核心流程稳定后叠加

### 9.2 关注点

- 审核团队的解析 Agent 和风险审查 Agent 是关键路径，需优先保证稳定性
- 对话团队的 QA Agent 分类准确度直接影响用户体验，需要迭代调优
- 三层记忆的一致性保证需要从第一天就做好（PG redo log 是核心）
- 可视化在后期叠加，但状态数据格式需在设计阶段对齐（Agent 输出格式一经确定后续改造成本高）
